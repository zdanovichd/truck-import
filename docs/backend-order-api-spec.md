# ТЗ: API создания заказа с сайта truck-import.ru (авторизованный пользователь)

Документ для бэкенда ЛК. Витрина (Next.js) после SSO хранит access token в httpOnly-cookie `ti_lk_token` и должна создавать заказ в ЛК от имени пользователя.

---

## 1. Контекст

- Витрина **truck-import.ru** после SSO получает **access token** ЛК и кладёт его в **httpOnly cookie** `ti_lk_token` (same-site, path `/`).
- Сейчас checkout на сайте отправляет данные на **`POST /api/orders`** на витрине; планируется **проксирование** на бэкенд ЛК с передачей этого токена.
- Публичный каталог ЛК доступен без токена (например, `GET …/api/v1/client/products`). Нужен **отдельный защищённый** метод создания заказа.

---

## 2. Цель

HTTP API на бэкенде ЛК:

- Принимает **Bearer access token** (тот же, что выдаётся после обмена кода SSO, например `POST …/api/v1/auth/sso/exchange`).
- Создаёт **заказ**, привязанный к пользователю/контрагенту в ЛК.
- Возвращает **идентификатор/номер заказа** и предсказуемые ошибки.

---

## 3. Авторизация

- Заголовок: **`Authorization: Bearer <access_token>`**.
- При отсутствии заголовка, невалидном или просроченном токене: **`401 Unauthorized`** с JSON-ошибкой (см. раздел 7).
- Без успешной авторизации заказ **не создавать**.

---

## 4. Метод и URL

Зафиксировать в ответе на ТЗ:

- Полный URL, например: `POST https://<lk-host>/api/v1/...`  
  **или**
- Путь относительно базы `LK_API_BASE_URL` (например `https://lk.truck-import.ru` без завершающего `/`).

Требование: один основной **`POST`** для «обычного» заказа с корзины витрины. Если архитектура иная (черновик → подтверждение) — описать цепочку шагов и URL каждого шага.

---

## 5. Тело запроса (JSON) — актуальный контракт

- **Владелец заказа** — только из **Bearer**; контакты в примечание к заказу — из профиля на ЛК, в тело **не** дублировать.
- **Обязательно:** `items` — непустой массив `{ "sku": "<как в products.sku>", "quantity": <1…999> }`; одинаковые `sku` на бэке схлопываются по количеству. Цены строк — из каталога на ЛК.
- **Опционально:** `comment` — строка комментария к заказу.

### Пример тела

```json
{
  "items": [
    { "sku": "YC25754", "quantity": 2 },
    { "sku": "17M-54-57520", "quantity": 1 }
  ],
  "comment": "Доставка до 18:00"
}
```

---

## 6. Успешный ответ

- HTTP **200** или **201**.
- JSON, минимум (имена полей зафиксировать):

```json
{
  "success": true,
  "order_id": "<uuid или int>",
  "order_number": "<человекочитаемый номер, если есть>"
}
```

Достаточно **одного** стабильного идентификатора для отображения пользователю на витрине.

---

## 7. Ошибки

Единый формат, например:

```json
{
  "success": false,
  "message": "Краткий текст для пользователя",
  "code": "VALIDATION_ERROR"
}
```

Рекомендуемые HTTP-коды:

| Код | Когда                                                |
| --- | ---------------------------------------------------- |
| 400 | Валидация (нет обязательных полей, пустая корзина)  |
| 401 | Нет / невалидный / просроченный токен               |
| 403 | Токен валиден, но нет права оформлять заказы        |
| 404 | Товар по `sku` не найден / недоступен (если так принято) |
| 409 | Конфликт (остатки, блокировка и т.п.) — по необходимости |
| 422 | Бизнес-ошибка ЛК (если отделены от 400)             |
| 5xx | Внутренняя ошибка сервера                           |

---

## 8. Идемпотентность (желательно)

Опционально: заголовок **`Idempotency-Key: <uuid>`** — при повторной отправке с тем же ключом не создавать дубликат заказа; описать поведение.

---

## 9. CORS

- Если витрина вызывает ЛК **только с сервера** (Next Route Handler) — CORS для браузера **не обязателен**.
- Если планируется **прямой** вызов из браузера — настроить CORS для origin витрины (`https://truck-import.ru`, при необходимости staging).

---

## 10. Что вернуть команде витрины после реализации

1. Точный **`POST` URL** эндпоинта.
2. **Пример `curl`** (без реального секрета; токен подставляют сами).
3. Финальная **JSON-схема** запроса и ответа (или фрагмент **OpenAPI**).
4. Таблица **кодов ошибок** и значений поля `code` (если есть).
5. Подтверждение: в каталоге для витрины в ответе есть **`sku`** (как в `products.sku`) для строк заказа.

---

## 11. Интеграция на витрине (реализовано)

**`POST /api/orders`** на сайте проксирует на ЛК:

- **URL ЛК:** `POST {LK_API_BASE_URL}/api/v1/client/orders`.
- **Токен:** `Authorization: Bearer` из httpOnly-cookie `ti_lk_token` — владелец заказа только из токена.
- **Тело:** `{ "items": [ { "sku": "<string>", "quantity": 1…999 } ], "comment"?: "…" }` — одинаковые `sku` на витрине схлопываются по количеству.
- **Успех:** `order_id`, `order_number` в ответе ЛК; витрина дублирует `orderNumber` для UI.
- **Ошибки ЛК:** статус и тело пробрасываются (`code`, `errors` при 422).

**Оформление в один клик:** «Оформить заказ» → `GET /auth/continue-checkout` → `GET /auth/place-order` (сбор `items` из cookie корзины: для каждой позиции подгружается товар, в заказ уходит **`sku`** из каталога) → редирект **`{LK_ADMIN_BASE_URL или LK_API_BASE_URL}/admin/orders/{order_id}`**. Опционально глобальный **`comment`**: **`LK_SITE_ORDER_COMMENT`** в `.env`.

URL **`/cart/checkout`** редиректит на **`/cart`**.

Код: `src/lib/lk-site-order.js`, `src/app/api/orders/route.js`, `src/app/auth/place-order/route.js`, `src/app/auth/continue-checkout/route.js`.
